# “每日 AI 论文与 AI 资讯 + 飞书私聊推送”Skill 设计稿

## 1. 目标定义

本文件不再把重点放在“项目实施方案”，而是明确：**要设计一个让 Codex 可稳定复用的 skill**，用于完成以下工作：

1. 初始化一个“每日 AI 论文与 AI 资讯推送”项目
2. 为项目生成或维护采集、筛选、摘要、推送、调度代码
3. 在本地或服务器环境中打通飞书私聊推送
4. 支持后续增量迭代，而不是每次都从零描述需求

核心目标不是“写一份项目书”，而是让另一个 Codex 在触发该 skill 后，能按稳定流程完成开发、调试、验证和扩展。

---

## 2. 现有方案的问题

原稿能作为项目立项文档，但**还不适合直接做成 skill**，主要有这几个问题：

### 2.1 文档定位不对

原稿偏“面向人”的实施书，内容完整，但缺少面向 Codex 的操作指导。  
skill 需要回答的是：

* 什么时候触发
* 触发后先做什么
* 哪些内容必须先确认
* 哪些代码结构是推荐默认值
* 哪些步骤必须验证

### 2.2 skill 边界不清

原稿混合了三层内容：

* 产品/项目目标
* 系统架构设计
* Codex 的开发任务

但没有明确 skill 到底负责：

* 只负责“生成代码”
* 还是也负责“部署与运行”
* 还是也负责“飞书权限打通”

skill 必须把边界写清，否则每次触发都会漂移。

### 2.3 对飞书 CLI 的角色描述不够实用

原稿强调“主链路用 Python 直调飞书 API，飞书 CLI 只做辅助”，这个原则是对的，但落地上还缺三点：

* 在当前你的实际环境里，`lark-cli` 已经可用，skill 应该优先利用现有能力，而不是抽象讨论
* 需要明确 bot 发送和 user 发送两种模式的差异
* 需要明确“开发验证阶段”和“生产运行阶段”分别推荐什么链路

### 2.4 缺少“skill 资源设计”

skill 不是只写一份 `SKILL.md`。它通常还需要配套资源：

* `references/`：架构、数据模型、飞书推送约定、提示词模板
* `scripts/`：初始化项目、发送测试消息、跑一次任务、验证配置
* `agents/openai.yaml`：给 UI 展示与默认提示

原稿没有把这些内容拆出来。

### 2.5 缺少“输入变量”和“默认决策”

skill 最怕每次都从头问用户。  
这个 skill 应该内置默认决策，例如：

* 默认语言：Python 3.11
* 默认存储：SQLite
* 默认推送类型：飞书私聊纯文本
* 默认论文源：arXiv `cs.AI` `cs.LG` `cs.CL` `stat.ML`
* 默认资讯源：OpenAI / Anthropic / DeepMind / Hugging Face Blog
* 默认时间：每天 09:00

如果用户不反对，skill 直接按默认值执行。

### 2.6 缺少验证闭环

原稿提了测试方案，但没有把验证流程写成 skill 的固定步骤。  
对这个 skill 来说，最低验收必须是：

1. 能抓到内容
2. 能生成摘要
3. 能渲染日报
4. 能发出飞书测试消息
5. 能记录历史避免重复发送

## 3. skill 的正确定义

### 3.1 skill 名称建议

`ai-daily-feishu-briefing`

### 3.2 skill 描述建议

用于创建、更新和排障“每日 AI 论文与 AI 资讯飞书私聊推送”项目。  
当用户需要：

* 初始化该类项目
* 接入或修改 arXiv / AI 官方资讯源
* 实现或修复飞书私聊发送
* 增加去重、筛选、摘要、调度逻辑
* 调整日报模板、推送规则、接收人配置
* 排查消息发不出去、重复推送、摘要质量差等问题

时，应触发此 skill。

### 3.3 skill 的职责边界

这个 skill 负责：

* 项目脚手架设计
* 数据源适配器实现
* 去重、排序、摘要、渲染逻辑实现
* 飞书发送打通与验证
* 定时任务接入
* 基础测试与排障

这个 skill 不默认负责：

* Web 管理后台
* 多租户系统
* 复杂推荐算法
* 完整运营平台

---

## 4. 推荐的系统实现边界

### 4.1 开发阶段

开发阶段建议优先使用：

* `lark-cli` 做飞书环境验证和测试发信
* Python 项目代码实现核心业务逻辑

原因：

* 你当前机器已经装好了 `lark-cli`
* 测试消息发送比直接写全量飞书 API 客户端更快验证
* skill 能更快闭环

### 4.2 生产阶段

生产阶段推荐分两种实现：

**方案 A：继续使用 `lark-cli` 发送**

适合：

* 运行环境就是你的常驻个人机器
* 已完成飞书 CLI 配置
* 更重视部署快，而不是最强通用性

**方案 B：Python 直连飞书开放平台 API**

适合：

* 准备部署到服务器或 CI
* 不希望运行时依赖本机 CLI 登录态
* 需要更明确的凭据和服务化部署方式

### 4.3 对这个 skill 的默认建议

**V1 默认走“双轨策略”**

* 验证阶段：优先用 `lark-cli`
* 产线阶段：默认保留切换到 Python API 的接口

这样更符合你当前环境，也避免一开始就把链路设计得过重。

---

## 5. skill 触发后的默认工作流

skill 被触发后，应按以下顺序工作：

1. 识别当前任务属于新建项目、增量开发、修复 bug、还是排障
2. 检查当前目录是否已有工程结构
3. 检查飞书相关能力是否可用
4. 明确使用 `lark-cli` 还是 Python API 作为当前阶段发送方式
5. 实现或更新数据采集模块
6. 实现或更新去重、打分、筛选模块
7. 实现或更新摘要生成模块
8. 实现或更新日报渲染模块
9. 打通一次真实测试发送
10. 验证历史去重和定时执行

如果任务只是“修发送失败”，则跳过无关模块，只聚焦推送链路。

## 7. 数据模型建议

### 7.1 原始内容模型

```python
from dataclasses import dataclass, field
from typing import List, Optional

@dataclass
class ContentItem:
    item_id: str
    item_type: str   # "paper" | "news"
    source: str
    title: str
    url: str
    published_at: str
    summary: str
    authors: List[str] = field(default_factory=list)
    tags: List[str] = field(default_factory=list)
    raw_text: Optional[str] = None
```

### 7.2 摘要结果模型

```python
from dataclasses import dataclass
from typing import List

@dataclass
class SummaryResult:
    brief: str
    highlights: List[str]
    why_it_matters: str
```

### 7.3 推送记录模型

```python
from dataclasses import dataclass

@dataclass
class PushRecord:
    item_id: str
    receiver_id: str
    channel: str
    status: str
    message_id: str | None = None
    pushed_at: str | None = None
```

## 8. 数据源默认策略

### 8.1 论文源

默认抓：

* arXiv `cs.AI`
* arXiv `cs.LG`
* arXiv `cs.CL`
* arXiv `stat.ML`

### 8.2 资讯源

默认抓：

* OpenAI Blog
* Anthropic News / Blog
* Google DeepMind Blog
* Hugging Face Blog

### 8.3 采集要求

每个源必须：

* 独立适配
* 有超时
* 有简单重试
* 字段缺失时可降级
* 输出统一 `ContentItem`

skill 应明确：**不要一开始做通用网页抓取框架。**

---

## 9. 内容处理默认策略

### 9.1 去重

按三层去重：

1. URL 去重
2. 标题归一化去重
3. 历史推送记录去重

### 9.2 打分

论文优先看：

* 新近发布时间
* 所属分类
* 关键词命中
* 是否明显与 agent、reasoning、multimodal、benchmark、inference 相关

资讯优先看：

* 是否官方源
* 是否新近发布时间
* 是否涉及模型发布、产品发布、开源发布
* 是否与 AI 应用落地和开发者工具相关

### 9.3 选取策略

V1 默认：

* Top 3 论文
* Top 3 资讯

支持在 `.env` 中配置。

---

## 10. 摘要生成默认策略

### 10.1 输出格式

始终输出 JSON，可解析，不允许自然语言散文式返回。

```json
{
  "brief": "2句话概括核心内容",
  "highlights": ["关键点1", "关键点2"],
  "why_it_matters": "为什么值得关注"
}
```

### 10.2 提示词原则

提示词要强调：

* 不空话
* 不逐字翻译
* 优先写信息增量
* 面向“产品经理 / 工程负责人 / AI 应用创业者”

### 10.3 降级策略

当摘要失败时：

* 保留原始标题
* 截断原始摘要
* 明确标记为“原始摘要降级结果”

这样日报不会因为单条摘要失败而整体失败。

## 11. 飞书发送设计

### 11.1 发送模式

skill 应支持两种发送实现：

**模式 A：CLI 发送**

调用类似：

```powershell
lark-cli im +messages-send --as bot --user-id "ou_xxx" --text "..."
```

**模式 B：API 发送**

流程：

1. 获取 tenant access token
2. 用 `open_id` 或 `user_id` 发送文本消息

### 11.2 默认优先级

默认优先级建议：

1. 若本机 `lark-cli` 可用且已授权，优先 CLI
2. 否则走 Python API
3. 两者都不可用时，明确报错并给修复建议

### 11.3 接收对象

V1 默认只支持单个接收人：

* `FEISHU_RECEIVER_OPEN_ID`

V2 再扩展多用户。

### 11.4 消息模板

V1 只做纯文本：

```text
AI 每日速递

【今日论文】
1. 标题
摘要：...
为什么值得关注：...
链接：...

【今日资讯】
1. 标题
摘要：...
为什么值得关注：...
链接：...
```

skill 应明确：V1 不优先做卡片消息。

## 15. skill 对 Codex 的执行要求

触发该 skill 后，Codex 应遵守这些规则：

1. 优先读取已有工程，不要盲目重建
2. 飞书能力先做最小验证，再做主流程开发
3. 新增数据源时，一个源一个适配器
4. 摘要输出必须结构化
5. 推送必须记录结果
6. 失败时优先降级，不要整批崩
7. 任何“每日任务”都必须可手动运行一次验证

---

## 16. 最低验收标准

skill 驱动完成的项目，至少要满足：

1. 能抓到论文和资讯
2. 能完成去重和筛选
3. 能生成中文摘要
4. 能渲染成可读日报
5. 能向指定飞书 `open_id` 发出测试消息
6. 能避免同一内容重复发送
7. 能手动运行一次
8. 能配置每日定时执行

## 18. 一句话定稿

这个需求不该继续写成“项目书”，而应该收敛成一个 **面向 Codex 的工程化 skill**：  
它的核心不是解释项目是什么，而是让 Codex 每次都能稳定地把“采集 -> 去重 -> 摘要 -> 飞书发送 -> 定时运行”这条链路做出来、修下去、跑起来。
