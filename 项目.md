# 每日飞书简报项目设计说明

## 目标

这个仓库的目标是把“信息抓取、日报生成、飞书发送、Codex 调度”拆成清晰的几层，既能作为一个可运行项目，也能作为一个可复用的 Codex skill 示例。

## 当前包含的能力

1. `ai_daily_push`
   负责抓取候选内容、去重、排序、渲染和发送
2. `skills/ai-daily-feishu-briefing`
   负责告诉 Codex 如何做这类日报任务
3. `codex_scheduler`
   负责把 Codex、skill、Windows 计划任务和飞书发送稳定串起来

## 为什么要分层

这样做的好处是：

- Skill 专注于“怎么思考和怎么组织内容”
- Python 项目专注于“怎么抓取、渲染和发送”
- Scheduler 专注于“怎么自动触发”

它们可以组合使用，也可以单独使用。

## 两种主要使用模式

### 1. Codex + skill 模式

适合你希望由 Codex 做最终选题和写作的场景。

流程：

1. `codex_scheduler/run_codex_briefing.ps1`
2. `codex exec`
3. `ai-daily-feishu-briefing`
4. 生成 UTF-8 日报文件
5. 通过稳定发送器发到飞书

### 2. 独立 Python 模式

适合你希望不依赖 Codex 运行时，直接用项目完成整条链路的场景。

流程：

1. `ai_daily_push/scripts/run_once.py`
2. 抓取候选内容
3. 去重和排序
4. 渲染日报
5. 发到飞书

## 后续可扩展方向

当前默认主题是 AI 论文和 AI 资讯，但这套模式可以扩展成更多主题，例如：

- 商业简报
- 金融简报
- 政策简报
- 科技产业简报

扩展时，优先改的是 skill 的说明、信息源和摘要规则，而不是发送基础设施。

## 公开仓库建议

公开发布时，README 应该主要面向使用者，说明：

- 这是什么
- 怎么运行
- 怎么安装 skill
- 怎么启动定时任务

而维护者视角的内容，例如哪些文件不要上传、发布前检查什么，应该放在单独文档里，比如 `GITHUB_PUBLISHING.md`。
